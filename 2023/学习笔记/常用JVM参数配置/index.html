常用JVM参数配置

发表于 2023-05-29 | 分类于 Java

内存参数

-Xms

初始化堆大小，例如-Xms16G

-Xmx

最大堆大小，例如-Xmx16G

一般初始化堆大小和最大堆大小设为一致，防止GC后需要重新计算分配内存

-Xss

设置每个线程的堆栈大小，例如-Xss512k

默认值：

  • Linux/ARM (32-bit)：320 KB
  • Linux/i386 (32-bit)：320 KB
  • Linux/x64 (64-bit)：1024 KB
  • OS X (64-bit)：1024 KB
  • Oracle Solaris/i386 (32-bit)：320 KB
  • Oracle Solaris/x64 (64-bit)：1024 KB
-Xmn

设置年轻代大小。

-XX:MaxDirectMemorySize=

最大堆外内存大小，默认不限制

-XX:NewRatio=

设置年轻代与老年代的大小比例，默认值为2，表示young:old为1比2，如果设为3则是1:3

-XX:SurvivorRatio=

设置 Eden 空间大小和 Survivor 空间大小之间的比例，默认值是 -XX:SurvivorRatio=8，表示 Eden : from : to = 8 : 1 : 1

-XX:MaxMetaspaceSize=

设置元空间最大大小

-XX:MetaspaceSize=

设置元空间初始化大小

-XX:MaxTenuringThreshold=

设置垃圾最大年龄。默认值15

内存溢出

-XX:+HeapDumpOnOutOfMemoryError

内存溢出时，生成dump文件

-XX:HeapDumpPath=

指定内存溢出时，生成的dump文件的位置，不加该参数时生成到该Java程序的目录

指定目录：-XX:HeapDumpPath=/home/zhouhj/

指定文件名：-XX:HeapDumpPath=/home/zhouhj/dump.hprof

-XX:OnOutOfMemoryError=

可以指定在内存溢出时，调用脚本。比如做重启应用等操作

-XX:OnOutOfMemoryError=/home/zhouhj/xxx.sh

-XX:+CrashOnOutOfMemoryError

当发生内存溢出的时候，JVM 就会退出，同时，JVM 会产生文本和二进制格式的崩溃日志（不推荐使用）

-XX:+ExitOnOutOfMemoryError

当发生内存溢出的时候，JVM会立马退出，且不打印日志（不推荐使用）

常用组合

GC日志

-XX:+PrintGC

打印GC基本信息

-XX:+PrintGCDetails

打印GC详细信息

-XX:+PrintTenuringDistrution

打印对象年龄分布情况，每个年龄的对象大小、个数信息

-XX:+PrintHeapAtGC

查看GC前后的堆、方法区可用容量变化

-XX:+PrintGCCause

产生GC的原因，默认开启

-XX:+PrintGCDateStamps

打印GC发生时的时间

-XX:+PrintGCTimeStamps

打印GC发生时，程序运行了多久

-XX:+UseGCLogFileRotation

开启GC日志分割

-XX:NumberOfGCLogFiles=

日志文件分割个数

-XX:GCLogFileSize=

单个日志文件大小

-Xloggc:

指定gc日志位置

-XX:+PrintGCApplicationStoppedTime

查看GC过程中用户线程停顿时间

-XX:+PrintReferenceGC

打印对象引用信息，强软弱虚

常用组合

垃圾收集器

G1

-XX:+UseG1GC

手动指定使用 G1 垃圾收集器执行内存回收任务（JDK1.8 需要手动指定，JDK1.9 以后默认）

-XX:G1HeapRegionSize=

设置每个 Region 的大小。值是 2 的幂，范围是 1～32MB，默认是堆内存的 1/2000。

-XX:MaxGCPauseMillis=

设置期望的每次年轻代垃圾回收最大 GC 停顿时间指标（JVM 会尽力实现，但不保证达到），默认是 200ms。

-XX:ConcGCThreads=

指定并发 GC 线程数，默认是 -XX:ParallelGCThreads/4，也就是在非 STW 期间的 GC 工作线程数。推荐ConcGCThreads = (3 + ParallelGCThreads) / 4

-XX:+ParallelGCThread=

设置 STW 工作线程数的值。当cpu小于等于8时，默认值为cpu逻辑核数。否则ParallelGCThreads= 8 + (N - 8) * 5 / 8

-XX:InitiatingHeapOccupancyPercent=

设置触发并发 GC 周期的 Java 堆占用率阈值。超过此值，就触发 GC。默认值是 45。高并发应用可以调低了一些，也就是提高了 GC 的频率，但是避免了垃圾对象过多，一次垃圾回收耗时过长的问题

-XX:G1ReservePercent=

在 G1 管理的老年代里预留 25% 的空闲内存，保证新生代对象晋升到老年代的时候有足够空间，避免老年代内存都满了，新生代有对象要进入老年代没有充足内存了。默认值是 10%，略微偏少，高并发应用可以调大了一些

-XX:G1OldCSetRegionThresholdPercent=

指定每轮 Mixed GC 回收的 Region 最大比例，默认 10%，也就是每轮 Mixed GC 附加的 Cset 的 Region 不超过全部 Region 的 10%，最多 10%，如果暂停时间短，那么可能会少于 10%。一般这个值不需要额外调整。

-XX:G1MixedGCCountTarget=

指定一个周期内触发 Mixed GC 最大次数，默认值 8。一次全局并发标记后，最多接着 8 次 Mixed GC，把全局并发标记阶段生成的 Cset 里的 Region 拆分为最多 8 部分，然后在每轮 Mixed GC 里收集一部分。这个值要和上一个参数配合使用，8*10%=80%，应该来说会大于每次标记阶段的 Cset 集合了。一般此参数也不需额外调整。

-XX:G1HeapWastePercent=

指定触发 Mixed GC 的堆垃圾占比，默认值 5%，也就是在全局标记结束后能够统计出所有 Cset 内可被回收的垃圾占整对的比例值，如果超过 5%，那么就会触发之后的多轮 Mixed GC，如果不超过，那么会在之后的某次 Young GC 中重新执行全局并发标记。可以尝试适当的调高此阈值，能够适当的降低 Mixed GC 的频率。

-XX:G1NewSizePercent=

G1新生代比例下限，默认值 5%。

-XX:G1MaxNewSizePercent=

G1新生代比例上限，默认值 60%。G1 会根据实际的 GC 情况 (主要是暂停时间) 来动态的调整新生代的大小，主要是 Eden Region 的个数

-XX:+ParallelRefProcEnabled

启用并行引用处理：如果你的应用程序中使用了大量的软引用和弱引用，那么启用并行引用处理能够更好地提高 GC 的效率；

-XX:-ResizePLAB

停止调整 PLAB 的大小，以避免 GC 线程之间的大量通信成本，以及每次 GC 期间的变化

CMS

-XX:+UseConcMarkSweepGC

手动指定老年代使用 CMS 收集器。

开启该参数后，会自动开启 -xx:+UseParNewGC。即年轻代用 ParNew，老年代用 CMS，老年代兜底用 Serial Old

-XX:ParallelcMSThreads =

设置 CMS 的线程数量。

默认启动的线程数是（ParallelGCThreads+3）/4，ParallelGCThreads 是年轻代并行收集器的线程数。
当 CPU 资源比较紧张时，受到 CMS 收集器线程的影响，应用程序的性能在垃圾回收阶段可能会非常糟糕。

-XX:CMSInitiatingOccupanyFraction=

设置堆内存使用率的阈值，一旦达到该阈值，便开始进行回收。

JDK5 及以前，默认值为 68，即当老年代的空间使用率达到 68% 时，会执行一次 CMS 回收；

JDK6 及之后，默认值为 92，即当老年代的空间使用率达到 92% 时，会执行一次 CMS 回收；

如果内存增长缓慢，可以设置较大的阀值，降低 CMS 的触发频率，减少老年代回收的次数，改善应用程序性能；

如果内存增长很快，应该降低这个阈值，避免频繁触发 Serial Old 收集器兜底，降低 Full GC 的执行次数。

-XX:+UseCMSCompactAtFullCollection

指定在执行完 Full GC 后对内存空间进行压缩整理

可以一定程度避免内存碎片的产生。

不过由于内存压缩整理过程无法并发执行，所带来的问题就是停顿时间变得更长了。

-XX:CMSFullGCsBeforeCompaction=

设置在执行多少次 Full GC 后对内存空间进行压缩整理。

Parallel

-XX:+UseParallelGC

手动指定年轻代使用Parallel并行收集器执行内存回收任务

-XX:+UseParallelOldGC

手动指定老年代使用Parallel并行收集器执行内存回收任务

jdk8默认开启，这个参数和上一个参数默认开启一个，另一个也会被开启

-XX:ParallelGCThreads=

设置年轻代并行收集器的线程数。一般最好与CPU数量相等，以避免过多的线程数影响垃圾收集性能

在默认情况下，当CPU数量小于8个，ParallelGCThreads的值等于CPU数量，当CPU数量大于8个，ParallelGCThreads的值等于3+[5*CPU_COUNT/8]

-XX:MaxGCPauseMillis=

设置垃圾收集器每次年轻代垃圾回收最大停顿时间，单位毫秒

为了尽可能的把停顿时间控制在该参数值以内，收集器在工作时会调整Java堆大小或其他一些参数

对于用户来说，STW的时间越短越好。但是在服务端，注重高并发，整体的吞吐量，所以服务器端适合Parllel

-XX:GCTimeRatio=

垃圾收集时间占总时间的比例(=1/(N+1))。用户衡量吞吐量的大小

取值范围(0,100)，默认99，即垃圾回收的时间不超过1%

和前一个参数-XX:MaxGCPauseMillis有一定的矛盾性。暂停时间越长，Radio参数越容易超过设定的比例

-XX:+UseAdaptiveSizePolicy

设置Parallel收集器具有自适应调节策略，在这种模式下，年轻代的大小、Eden和Survivor的比例、晋升老年代的对象年龄等参数会被自动调整，以达到在堆大小、吞吐量和停顿时间之间的平衡点

在手动调优比较困难的场合，可以直接使用这种自适应的方式，仅指定虚拟机的最大堆、目标的吞吐量（GCTimeRatio）和停顿时间（MaxGCPauseMills），让虚拟机自己完成调优工作
